## Phase F — Close phase-E follow-ups (5 days, critical)

Fix the 5 items in `docs/specs/plans/phase-e-followups.md § Remaining
open work`. These are explicit known-broken paths.

### F.1 LLM tool loop tx propagation (2 days, CRITICAL)

**Problem.** `lib/tools.ts::executeTool`, `lib/memory.ts`,
`lib/memory-compaction.ts`, and `lib/llm-call.ts::getProviderKey`
still call `getDb()` inside `runConversationalTurn`'s transaction.
Tests pass only because test agents have no tools configured.
**Production agents with tools will hit FORCE RLS rejection.**

**Fix pattern.** Same as the billing helpers in v1.2: add an optional
`tx?: WsDb` arg, default to `getDb()` when omitted. Mechanical change.

**Tasks:**

1. Audit every call site of `executeTool`, `getProviderKey`, memory
   reads/writes inside `runConversationalTurn`. List in a tracking
   issue.
2. Thread `tx` through each function signature. Update internal
   callers.
3. Write integration test: agent with `agent_handoff`,
   `knowledge_search`, `memory_set`, `mnemosyne_remember` tools must
   complete a turn end-to-end under FORCE RLS without leaking outside
   its workspace.
4. Verify against testcontainer (existing pattern from
   `tests/integration/channels/router.spec.ts`).

**Done when:** new integration suite passes; existing tests still pass;
manual smoke test against `acme-inc` with a tool-using agent runs clean.

### F.2 flow-engine inline branch tx (0.5 days)

**Problem.** `lib/flow-engine.ts::executeFlow` inline (synchronous)
branch uses `getDb()`. Only hit for `agent.kind === "flow"` with short
flows. Async branch is safe (workers carry their own envelope).

**Fix.** Thread `tx?: WsDb` into the inline branch.

**Done when:** unit test exercises both branches; flow-using agent with
`kind === "flow"` runs end-to-end.

### F.3 GDPR exporter credentials redaction (1 day, SECURITY)

**Problem.** `agents`, `conversations`, `messages`, `knowledge`
exporters dump full rows. Possible leakage of `encryptedCredentials`,
`apiKey`, prompts containing customer secrets, etc.

**Tasks:**

1. Audit each exporter: enumerate every column dumped.
2. Define redaction policy per table:
   - `ai_provider.api_key` → never include
   - `agent.config.api_key` (if any inline) → strip
   - `message.text` → keep (user data is the point of export)
   - `tool_call.result` → review case by case
3. Implement column-level redaction in `lib/gdpr/exporters/*`.
4. Add snapshot test: export a fixture workspace, grep the resulting
   zip for `sk-`, `xoxb-`, `pk_live_`, etc. → must be empty.

**Done when:** snapshot test asserts no secret patterns in any export;
ADR documents the redaction policy.

### F.4 Filesystem download HTTP route (0.5 days)

**Problem.** When `STORAGE_BACKEND=filesystem`, the export URL is
`file://...` which a browser can't open. Self-host has broken
download links.

**Fix.**

1. Create `apps/web/app/api/exports/[token]/route.ts` that:
   - Verifies an HMAC-signed `(token, expiry)` tuple.
   - Streams the file from `GDPR_EXPORT_DIR` with proper
     Content-Disposition.
2. Update the filesystem storage adapter to mint these URLs instead
   of `file://`.
3. Add expiry (default 24h) + single-use token (rotate after first
   download? optional).

**Done when:** self-host GDPR export → click download → file streams
to browser. E2E test with `STORAGE_BACKEND=filesystem`.

### F.5 GDPR true streaming (1 day)

**Problem.** `archiver` is configured but the result buffers in
memory before upload. Breaks for workspaces > 100MB.

**Fix.** Pipe `archiver` directly into:

- S3 multipart upload (use `@aws-sdk/lib-storage`'s `Upload` class
  which handles multipart automatically).
- Or filesystem write stream for self-host.

**Done when:** export a 500MB synthetic workspace, peak RSS during
export stays under 200MB (vs. >500MB before).

## Phase G — Test reinforcement (3 days)

The codebase has decent unit coverage but the **cross-tenant
isolation guarantees** are mostly enforced by RLS at runtime — there
are no automated tests that prove a regression won't leak data.

### G.1 Tenant isolation matrix (1.5 days)

A test suite that creates two workspaces (`A` + `B`) with disjoint
data and asserts that:

1. Every API route returning a list, when called with workspace `A`'s
   session, never includes workspace `B`'s rows.
2. Every API route doing point reads (e.g. `GET /api/agents/[id]`)
   returns 404 when given a foreign id.
3. Every API route doing writes (POST/PATCH/DELETE) rejects foreign
   ids with 404 (not 403 — non-enumerable).
4. The Mnemosyne tables (`mnemo_fact`, `mnemo_entity`, etc.) honor
   `actor_id` isolation in addition to `workspace_id`.
5. The MCP server respects workspace boundaries when called with
   `Authorization: Bearer <key from workspace A>`.

Build this as a parameterized test that enumerates every API route
from `apps/web/app/api/` and runs the same assertions against each.

### G.2 Phase F regression tests (1 day)

Tests for F.1–F.5: tool loop tx, flow inline branch, GDPR redaction,
filesystem download, streaming export.

### G.3 E2E Playwright critical flows (0.5 days, optional)

Convert the manual UI audit into automated E2E:

- Sign up → create workspace → invite member → create agent → send
  message → see conversation → check cost.
- Upload doc to KB → semantic search returns hit.
- Flow builder: drag node → connect → save → run.

Keep narrow — 3 flows that cover the demo path. More can come later.

Phase 0.2 of the tenant-hardening plan was supposed to capture a
baseline. Verify it exists; if not, capture one now.

### H.1 Measure (0.5 days)

For each top-traffic route, capture under realistic load (k6 / autocannon):

- `GET /api/conversations?limit=50`
- `GET /api/mnemo/facts?status=active&limit=25`
- `POST /api/chat/turn` (with tools)
- `POST /api/mnemo/recall`

Record p50, p95, p99, throughput, error rate. Commit to
`docs/perf/baseline-2026-05-26.md`.

### H.2 Index review (0.5 days)

Run `EXPLAIN ANALYZE` on the queries behind each route. List any seq
scans on >10k-row tables. Add missing indexes.

### H.3 N+1 audit (1 day)

Scan all React Server Components + API routes for N+1 query
patterns. Particularly likely in:

- Conversations list (join messages count)
- Agents list (join team + channel count)
- Knowledge bases list (join doc count + chunk count)

Replace with proper joins or batch loaders.

## Phase I — Production deployment (2–3 days)

Self-host today is `docker compose up -d`. v1.0 needs more:

### I.1 Docker production image (0.5 days)

- Multi-stage build: builder stage with full deps, runtime with only
  pruned production deps.
- Non-root user.
- Healthcheck endpoint.
- Cut image size below 300MB.

### I.2 Helm chart for k8s (1 day)

A minimal helm chart that deploys:

- Web (3 replicas behind a Service)
- Worker (2 replicas)
- Postgres external (operator provides URL)
- Migrations as a Job that runs pre-install/pre-upgrade.

### I.3 Vercel + Fly deploy buttons (0.5 days)

README "Deploy to Vercel" + "Deploy to Fly" buttons that hit a
working template with required env vars pre-declared.

### I.4 Health checks + readiness (0.5 days)

- `GET /api/health` — liveness (always 200 unless catastrophic)
- `GET /api/ready` — readiness (200 only when DB reachable + worker
  heartbeat within last 30s)
- Document the difference in `docs/runbooks/health-checks.md`.

## Phase L — First-run UX (3 days)

A first-time operator landing on a fresh self-host has no agents, no
flows, no channels. Today they see empty states. They should see a
**guided tour**.

### L.1 Onboarding checklist (1 day)

A persistent widget in the topbar that tracks:

- [ ] Connect a provider (Anthropic/OpenAI)
- [ ] Create your first agent
- [ ] Connect a channel
- [ ] Send your first message
- [ ] Create a knowledge base
- [ ] Invite a teammate

Dismissable once all items checked. Driven by reading existing DB
state (no separate onboarding table).

### L.2 First-run sample data (0.5 days)

After signup, if `enableSampleData=true` (default for self-host
demos), seed the new workspace with 1 example agent ("Demo Bot") and
1 example flow ("Hello Flow"). Operator can delete with a button.

### L.3 Empty-state polish (0.5 days)

Replace every "No data" empty state with a teaching one — 1-line
explanation + a CTA button to create the first item. Already partly
done; this is a sweep to catch the rest.

### L.4 Mobile responsiveness audit (1 day)

The audit was done at desktop res. Sweep all main pages at
375x812 (iPhone 13). Top issues likely: topbar overflow, sidebar
collapse, conversation panel, flow builder (likely won't work on
mobile; document that).

## Open questions (need decisions before kickoff)

1. **Cloud vs. self-host first?** Phase I has work for both — should
   we ship Docker-compose-only as v1.0 and do helm/Vercel later, or
   block on all three?
2. **Test coverage threshold?** What % of files / lines / branches
   should be enforced in CI for v1.0?
3. **Mobile support level?** "Read-only on mobile, edit on desktop"
   is the simplest contract. Anything more is a separate workstream.
4. **Telemetry default opt-in?** Do we ship anonymous usage telemetry
   on by default with an opt-out, or off by default with an opt-in?
   (Affects both product analytics and Sentry.)
5. **Cut a v0.9 first?** Several of these phases (J, K, L) are
   polish. A v0.9 release after Phase F+G might be the right
   timing — gets the correctness fixes out the door fast, then v1.0
   is the polish round.
